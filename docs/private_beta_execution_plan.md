# Private Beta Execution Plan

## Decision

The next go-live path is `private beta first`, not live checkout first.

## Why this path wins now

- auth exists
- upload exists
- Today / Review / Brain exist
- billing boundary exists
- launch ops discipline exists

What is still missing is not foundation.
It is real-user validation of credibility, reliability, and support load.

## Beta Objective

Prove that a small invited user can:

1. sign in,
2. upload a GG packet,
3. receive believable outputs,
4. understand what to do next,
5. and get support when something breaks.

## Invite Shape

- 1 to 3 users first
- high-context users only
- direct support path available
- no public announcement

## Success Criteria

- signup/login success
- upload success
- duplicate handling success
- Today output understood
- Review / Brain feel believable enough to continue using
- support burden is manageable

## Failure Criteria

- uploads fail unpredictably
- outputs feel generic/unbelievable
- users do not understand what Today / Review / Brain mean
- support issues cannot be triaged quickly

## Recommended Execution Order

1. choose first invite list
2. verify environment and support path
3. run one internal dry run
4. invite first beta user
5. capture feedback
6. decide whether checkout should come next
